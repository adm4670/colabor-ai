1. Visao Geral

O colabor-ai e uma plataforma open-source de orquestracao de agentes de IA. Multiplos agentes especializados colaboram coordenados por um Orchestrator central, com suporte a ferramentas, memoria de longo prazo, planejamento multi-step e streaming de eventos.

TypeScript Node.js DeepSeek / OpenAI Telegram Bot Electron React + Vite Jest Puppeteer node-cron
CANAIS DE ENTRADA Telegram | Terminal | Web | API \ | / / \ | / / v v v v +-----------------------------------+ | ORCHESTRATOR | | (core/orchestrator/) | | | | [Planner] --> [Router] --> [Critic] | | | | | | +----- Agent Selection ----+ | | | +--------------+--------------+ | | | | | Assistant PythonAgent Browser | | | | | +---- Tools (Python, Web, Shell, etc) | | | +--------------+--------------+ | | LLM API (DeepSeek/OpenAI) | +-----------------------------------+ | +------------+------------+ | | | MEMORY CONTEXT PLAN (longo (token (multi- prazo) budget) step)

O sistema opera em tres camadas:

CamadaDiretorioResponsabilidade

Cloudcloud/Servidor REST API, autenticacao, endpoints de chat

2. Fluxo de Execucao

Cada mensagem do usuario passa pelo seguinte pipeline no Orchestrator:

1. Recepcao:     Usuario envia mensagem (Telegram/Terminal)
    2. Rate Limit:   Verifica se a sessao nao excedeu o limite de mensagens
    3. Transcript:   Carrega o historico da sessao corrente (.jsonl)
    4. Contexto:     ContextEngine monta o contexto com token budget
    5. Memoria:      MemoryEngine busca fatos relevantes (MEMORY.md + notas diarias)
    6. Skills:       SkillsManager carrega skills relevantes ao contexto
    7. Hook before_planner:  Executa hooks registrados
    8. Planner:      PlannerAgent decide qual agente usar e com qual instrucao
    9. Agent Loop:   O agente escolhido executa (LLM + tools + sub-agents)
    10. Reflector:   ReflectorAgent avalia se o resultado e satisfatorio
    11. Hook before_response: Hook de pre-resposta
    12. Resposta:    Resposta final enviada ao usuario (com formatacao)
    13. Consolidacao: Transcript salvo, DreamTask agenda consolidacao de memoria

4. Orchestrator (Maestro)

O AgentOrchestrator (core/orchestrator/orchestrator.ts) e o componente central que coordena todos os agentes. Ele implementa o loop principal:

4.1 Responsabilidades

  • Rate Limiting: Protecao contra uso excessivo por sessao
  • Transcript: Carrega e persiste historico JSONL
  • Contexto: Integracao com ContextEngine para token budget
  • Memoria: Busca fatos relevantes via MemoryEngine
  • Skills: Carrega skills relevantes ao contexto
  • Hooks: Executa hooks em pontos-chave do pipeline
  • Permissoes: Verifica permissoes antes de executar tools
  • Planejamento: Consulta o PlanManager para seguir planos ativos
  • Sub-agentes: Spawna sub-agentes via SubAgentRunner
  • Background Tasks: Cria e monitora tarefas assincronas
  • Streaming: Emite eventos via EventStream

4.2 Metodo Principal: run()

async run(input: string, sessionId: string, userId: number): Promise<string> {
      // 1. Rate limit check
      // 2. Carrega transcript e restaura estado
      // 3. Constroi contexto (ContextEngine)
      // 4. Busca memoria relevante (MemoryEngine)
      // 5. Carrega skills (SkillsManager)
      // 6. Verifica plano ativo (PlanManager)
      // 7. Loop de agentes:
      //    a. Planner decide agente + instrucao
      //    b. Hook before_agent
      //    c. Agente executa (Agent.run com tools)
      //    d. Hook after_agent
      //    e. Reflector avalia
      //    f. Se precisa mais -> volta ao planner
      // 8. Hook before_response
      // 9. Retorna resposta final
    }

5. Agentes

Cada agente e uma instancia da classe Agent (core/agent/agent.ts) com configuracao especifica de nome, role, goal, backstory, tools e functions. Sao registrados no AgentRegistry (core/agents/agent-registry.ts).

5.1 Classe Agent

class Agent {
      name: string;           // Nome unico
      role: string;           // Papel no sistema
      goal: string;           // Objetivo principal
      backstory: string;      // Personalidade/contexto
      model: string;          // Modelo LLM (default: deepseek-chat)
      tools: any[];           // Ferramentas OpenAI-format
      functions: Record<string, Function>;  // Handlers
    
      async run(instruction: string): Promise<string>  // Executa uma instrucao
      resetHistory(): void                              // Limpa historico
    }
    
    // Suporte a tool calling com loop automatico:
    // 1. Envia prompt + tools ao LLM
    // 2. Se LLM retorna tool_calls, executa handlers
    // 3. Adiciona resultados como tool messages
    // 4. Reenvia ao LLM para continuar
    // 5. Repete ate obter resposta final (max 10 iteracoes)

6. Ferramentas (Tools)

Cada ferramenta implementa a interface ToolDefinition e e registrada no ToolRegistry. As ferramentas sao expostas ao LLM no formato OpenAI function calling.

6.1 Interface ToolDefinition

interface ToolDefinition<TArgs, TResult> {
      name: string;                    // Nome unico
      description: string;             // Descricao para o LLM
      parameters?: {                   // Schema JSON dos parametros
        type: "object";
        properties?: Record<string, any>;
        required?: string[];
      };
      execute: (args: TArgs, context: ToolContext) => Promise<TResult> | TResult;
    }
    
    interface ToolContext {
      agentName: string;
      userId: number;
    }

7.2 ContextEngine Contexto

Gerencia o orcamento de tokens do historico. Usa estrategia hibrida de 3 zonas:

  • Zona 1 (recente): Ultimas N mensagens mantidas intactas
  • Zona 2 (sumarizacao): Mensagens intermediarias sumarizadas via LLM
  • Zona 3 (descarte): Mensagens antigas removidas
class ContextEngine {
      compact(messages, systemPrompt): ContextSummary
      estimateTokens(text): number       // Via tiktoken
    }
    
    // Config:
    // maxTokens: 8000, recentRatio: 0.6, minMessages: 6
    // keepRecentIntact: 5, summarizeZoneSize: 10

7.3 MemoryEngine Memoria

Sistema de memoria de longo prazo com tres capacidades:

  1. recall(): Busca semantica com scoring de relevancia (MEMORY.md + notas diarias)
  2. consolidate(): Extrai fatos, decisoes, preferencias do transcript
  3. manageWorkingMemory(): Compacta historico preservando contexto critico
class MemoryEngine {
      recall(query, context?, maxResults?): string
      consolidate(sessionId, messages): ConsolidatedFact[]
      manageWorkingMemory(messages): ContextMessage[]
    }

7.5 EventStream Streaming

AsyncIterable que permite emitir eventos em tempo real durante a execucao. Baseado no padrao produtor-consumidor.

type StreamEventType =
      | "agent_start" | "agent_end"
      | "turn_start" | "turn_end"
      | "message_start" | "message_update" | "message_end"
      | "text_start" | "text_delta" | "text_end"
      | "tool_call_start" | "tool_call_delta" | "tool_call_end"
      | "tool_execution_start" | "tool_execution_update" | "tool_execution_end"
      | "progress";
    
    class EventStream<T, R> {
      push(event: T): void        // Emite um evento
      end(result?: R): void       // Finaliza o stream
      [Symbol.asyncIterator]()    // Consumo via for-await-of
    }

8. Cloud Layer

O diretorio cloud/ contem um servidor REST API completo com:

Express Server

Servidor HTTP com WebSocket para comunicacao em tempo real (server.ts).

Autenticacao

Rotas de auth com middleware de verificacao (routes/).

Tool Protocol

Protocolo de comunicacao cloud <-> client para execucao remota de ferramentas (protocol/).

Testes

Suite de testes Jest: auth, context-engine, tool-protocol (__tests__/).

O cloud layer compartilha a mesma arquitetura de agentes e ferramentas do core, mas expoe via API REST em vez de Telegram/Terminal.

9. Client Layer

Aplicacao desktop construida com:

  • Electron: Shell nativa (main.ts, preload.ts)
  • React + Vite: Interface (App.tsx, componentes)
  • TypeScript: Tipagem estatica

O client se conecta ao cloud layer via WebSocket para comunicacao em tempo real.
