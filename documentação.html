🔭 Visão Geral da Arquitetura

O colabor-ai é um orquestrador multi-agente construído em TypeScript que coordena agentes especializados de IA para executar tarefas complexas. Inspirado no paradigma Planner-Worker, o sistema recebe uma requisição do usuário, planeja a execução, delega para agentes especializados, reflete sobre os resultados, e entrega a resposta final.

🏗️ Arquitetura Detalhada

2.1 Orquestrador (AgentOrchestrator)

O AgentOrchestrator implementa um loop de execução que:

  1. Recebe o input do usuário
  2. Chama o PlannerAgent para decidir o próximo agente
  3. Executa o agente selecionado com a instrução do planner
  4. Chama o ReflectorAgent para avaliar o resultado
  5. Se o resultado for incompleto, tenta novamente com abordagem diferente
  6. Repete até o planner retornar "finish"

Possui rate limiting (default: 100 mensagens/sessão), EventStream para telemetria em tempo real, e ContextEngine para gestão de tokens.

2.2 Sistema de Memória (SOTA)

🧠

Working Memory

Memória de curto prazo com orçamento de tokens (8k-16k). Mantém plano ativo, restrições, e entradas com importância.

📖

Memória Episódica

Sumários de interações completas com embeddings TF-IDF. Busca por similaridade semântica. Limite: 1000 memórias.

🔗

Memória Semântica

Knowledge Graph no formato sujeito-predicado-objeto. Ex: usuário → prefere → dark mode

⚙️

Memória Procedural

Armazena planos bem-sucedidos: objetivo, passos executados, resultado para reúso futuro.

🧹

Refresh & Forget

Meia-vida dinâmica para memórias. Refresh a cada 20 passos. Esquecimento após 3 ciclos sem acesso.

🎯

Atenção Seletiva & Foco

Filtra working memory antes de cada LLM: remove triviais, prioriza plano. Detecta divagação automaticamente.

⚠️ Nota: Existem dois sistemas de memória coexistindo: o memory-engine.ts (antigo, usado pelo orchestrator atualmente) e o memory-engine-sota.ts (novo SOTA, completo, mas não integrado ativamente).

2.3 Sistema de Skills

Skills são instruções Markdown com frontmatter YAML na pasta skills/. O SkillsManager carrega skills relevantes via matching de palavras-chave.

💬

communication.md

Diretrizes de comunicação para diferentes canais (Telegram, chat, etc.)

🐍

python-dev.md

Boas práticas de Python: PEP 8, tratamento de erros, performance

🌐

web-search.md

Instruções para busca na web usando o BrowserAgent

2.4 Telemetria

Sistema de observabilidade que coleta tokens por chamada LLM, chamadas de agentes, decisões do planner, ferramentas executadas, reflexões e durações. Relatórios em JSON na pasta telemetry/. Comandos Telegram: /telemetry e /status.

🔧 Pontos de Melhoria Identificados

🔴

3.1 Segurança

3 itens
API Keys hardcoded nos agentes Médio

Todos os 6 agentes repetem apiKey: process.env.DEEPSEEK_API_KEY || "". Código duplicado e propenso a erro.

Sugestão: Criar uma factory centralizada (createAgent()) que injeta a API key automaticamente.

PythonExecTool sem sandbox Crítico

Executa código Python arbitrário via child_process.exec() sem container, sem restrição de recursos.

Sugestão: Executar em container Docker descartável ou usar Pyodide/vm2.

Arquivo .env no repositório Crítico

O arquivo .env com credenciais reais está presente no disco.

Sugestão: Verificar .gitignore e considerar usar secrets manager (dotenv-vault, 1Password CLI).

3.3 Potenciais Bugs

5 itens
JSON.parse sem validação de schema Alto — Se o LLM devolver JSON inválido, o sistema assume sucesso parcial silenciosamente.
Rate limit em memória RAM apenas Médio — messageCounts: Map é volátil. Se o processo reiniciar, perde tudo.
Token counting heurístico Médio — content.length / 4 é impreciso. Usar tiktoken.
Regex quebrado no compressor Alto — /vamoss+w+[^.]*./gi mal formatado, nunca captura decisões.
Conversations map em RAM no Telegram Médio — Histórico não persistido entre reinícios.

🧠 Sugestões para Arquitetura Mais Inteligente sem Perder Desempenho

Seis melhorias que aumentam a inteligência do sistema ao mesmo tempo que reduzem ou mantêm a latência:

🔄

Roteamento Preditivo de Agentes

Problema: Planner consulta LLM a cada requisição mesmo para tarefas óbvias.

Solução: Classificador leve (TF-IDF + regras + cache) que aprende padrões de roteamento.

+Inteligência+Performance
⚡

Cache Semântico de Resultados

Problema: Perguntas similares geram chamadas LLM idênticas.

Solução: Cache baseado em embedding da pergunta com threshold >95% de similaridade.

+Inteligência+Performance
🔀

Execução Paralela Seletiva

Problema: Orquestrador processa tudo em série.

Solução: Classificar tarefas como independentes (paraleliza) ou sequenciais (executa em série).

+Inteligência+Performance
📝

Auto-aprendizado via Reflector

Problema: Reflector só avalia se funcionou, sem aprendizado.

Solução: Alimenta banco de "lições aprendidas" que o Planner consulta.

+InteligênciaNeutro
🗄️

Memória Unificada Hierárquica

Problema: Dois sistemas de memória sem comunicação.

Solução: Store hierárquico: quente (RAM) → morno (JSON) → frio (persistente), com TTL diferente.

+Inteligência+Performance
🧩

Lazy Loading de Skills

Problema: Todas as skills carregadas no system prompt.

Solução: Carregar skills sob demanda baseado na intenção detectada.

+Inteligência+Performance
