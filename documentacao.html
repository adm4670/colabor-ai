🏗️ Padrão Arquitetural

O Planner-Worker é o padrão central: um Orquestrador recebe a mensagem, o Planejador decide qual agente chamar (browser, python, shell, etc.), e o resultado volta em loop até finalizar com uma resposta consolidada.

Exemplo de Código

Como um task flui pelo sistema de orquestração

    // Exemplo simplificado do fluxo de orquestração no colabor-ai
    
    import { AgentRegistry } from './agents/agent-registry';
    import { Orchestrator } from './orchestrator/orchestrator';
    import { MemoryEngine } from './memory/memory-engine';
    
    // 1. Configuração
    const registry = new AgentRegistry();
    const memory = new MemoryEngine();
    const orchestrator = new Orchestrator(registry, memory);
    
    // 2. Registrar agentes especializados
    registry.register(plannerAgent);
    registry.register(browserAgent);
    registry.register(pythonAgent);
    registry.register(shellAgent);
    registry.register(answerAgent);
    
    // 3. Receber mensagem do usuário (via Telegram)
    const userMessage = "Pesquise o clima em São Paulo e salve em um arquivo";
    
    // 4. Orquestrador processa
    const result = await orchestrator.run({
        input: userMessage,
        sessionId: "session_123",
    });
    
    // 5. Internamente o Orchestrator:
    //    a) Chama o PlannerAgent para decidir o plano
    //    b) Planner: ["browser (pesquisar clima)", "python (processar)", "shell (salvar)"]
    //    c) Executa cada passo em sequência
    //    d) Reflector valida resultados parciais
    //    e) AnswerAgent consolida resposta final
    
    // 6. Resposta é enviada via stream de eventos
    eventStream.emit({
        type: "response",
        content: result,
        sessionId: "session_123",
    });

Melhorias

Sugestões baseadas na análise estática do código-fonte

🔴

Testar o módulo de Memória SOTA

Alta Prioridade

O módulo de memória possui 14 arquivos (AttentionFilter, KnowledgeGraph, EmbeddingService, MemoryEngine, etc.) e zero testes. É o módulo mais complexo e crítico — qualquer regressão aqui afeta toda a orquestração do sistema.

attention-filter.ts memory-engine.ts knowledge-graph.ts embedding-service.ts
🔴

Refatorar arquivos grandes

Alta Prioridade

orchestrator.ts (503 linhas de código) e telegram.ts (311 linhas) são muito densos. Idealmente, um arquivo deve ter no máximo 300-400 linhas. Quebrar em módulos menores melhora manutenibilidade e testabilidade.

Extrair handlers Separar rotas Modularizar lógica
🟡

Resolver dívida técnica (24 TODOs)

Média Prioridade

24 itens de TODO/FIXME/HACK espalhados por 13 arquivos. Alguns são simples (documentação de métodos privados), outros podem indicar bugs ou decisões pendentes. Agendar uma sessão de cleanup é recomendado.

🟡

Adicionar JSDoc ao código

Média Prioridade

A taxa de comentários é de apenas 10%. Módulos críticos como memory-engine.ts, orchestrator.ts e telegram.ts carecem de documentação JSDoc em classes e métodos públicos, dificultando a onboarding de novos desenvolvedores.

🟢

Modularizar diretório raiz

Baixa Prioridade

O diretório raiz contém 12.370 linhas. Organizar documentação em subdiretórios ou mover arquivos soltos (como analise-run-method.md, diagnostico-orchestrator.md) para docs/ traria mais organização.

Glossário

Planner-Worker

Padrão arquitetural onde um Planner decide o plano e Workers executam as tarefas.

SOTA

State of the Art — sistema de memória de última geração com AttentionFilter, KnowledgeGraph, EmbeddingService.

AgentRegistry

Registro central que mapeia nomes de agentes para suas implementações.

Stream de Eventos

Canal de comunicação em tempo real para emitir progresso, resultados e logs.

Telemetry Dashboard

Painel em tempo real mostrando tokens consumidos, eventos e performance dos agentes.

Orchestrator

Componente central que recebe mensagens, coordena agentes e gerencia o ciclo de vida das tarefas.
