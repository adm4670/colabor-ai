1 Visao Geral

Proposito

O colabor-ai e uma plataforma para criar e executar agentes de IA que colaboram para resolver tarefas. Interpreta solicitacoes, raciocina sobre tarefas, executa acoes e integra diferentes ferramentas.

Stack Tecnologica

  • Node.js + TypeScript (strict mode)
  • OpenAI API (GPT-4o-mini, GPT-5-nano)
  • DeepSeek API (deepseek-v4-pro)
  • Telegram Bot API / MongoDB (opcional)
  • Puppeteer / Jest + Babel

Componentes

  • Controller — ponto de entrada (API/Telegram/CLI)
  • Orchestrator — coordena agentes
  • Agent — orquestra raciocinio + acoes
  • LLM Service — comunicacao com modelos
  • Tools — ferramentas executaveis

Fluxo de Execucao

1
Usuario envia mensagem pelo canal (Console ou Telegram)
2
Orchestrator recebe e cria contexto inicial
3
PlannerAgent decide qual agente executar
Retorna JSON: { "agent": "nome", "instruction": "..." }
4
SubAgent executa a tarefa (Python, Browser, etc.)
5
Loop repete se planner retornar "finish"
Max 10 steps, com protecao contra loop infinito
6
Resposta retornada ao usuario + nota salva em memory/

3 Detalhamento de Cada Arquivo

NUCLEO core/agent/agent.ts

Agent (Classe Base)

Classe principal que representa um agente. Orquestra o raciocinio e as acoes: decide se responde diretamente ou executa tools. Usa OpenAI SDK para comunicacao com LLM (DeepSeek por padrao).

Importancia: Todo agente do sistema herda desta classe. Gerencia historico, tools, functions e system prompt. [MELHORADO] buildSystemPrompt() agora carrega Skills + Notas Diarias
CANAL core/agent/main.ts

Console Agent (CLI)

Ponto de entrada por linha de comando. Cria instancias de Agent para o Orchestrator e inicia um loop readline.

Problema: Recria agentes manualmente com new Agent(...) em vez de reutilizar os de core/agents/.
AGENTES core/agents/*.agent.ts

7 Agentes Especializados

  • PlannerAgent — Cerebro do sistema, decide qual agente executar
  • AssistantAgent — Conversa geral, perguntas simples
  • PythonAgent — Executa codigo Python
  • BrowserAgent — Automacao web (Puppeteer)
  • ShellAgent — Comandos shell (npm, git, fs)
  • TaskManagerAgent — CRUD de tarefas
  • Answer/WriterAgent — Formata resposta final
Problema: Todos sao singletons (instancia unica). Historico vaza entre sessoes de usuarios diferentes!
NOVO core/context/context-engine.ts

ContextEngine

Gerencia o orcamento de tokens do historico de conversa. Comprime mensagens antigas quando necessario para caber no limite.

Status: 70% — Modo trim funcional (corta), mas modo summarize eh STUB: "implementacao futura. Por enquanto, usa trim como fallback"
MELHORADO core/memory/memory_search.ts

Memoria de Longo Prazo

Sistema de memoria em 3 camadas: MEMORY.md (fatos), notas diarias (memory/YYYY-MM-DD.md), e busca textual.

  • saveDailyNote() — salva resumo ao final da sessao
  • loadRecentNotes() — carrega notas de hoje + ontem
  • memorySearch() — busca textual por palavras-chave
  • appendToMemory() — adiciona ao MEMORY.md
Status: 100% funcional! Orchestrator salva notas, Agent carrega no system prompt. Memoria real entre dias.
CENTRAL core/orchestrator/orchestrator.ts

Orchestrator

Cerebro do sistema. Recebe input do usuario, consulta o PlannerAgent, executa o agente escolhido, e itera ate "finish".

  • Rate limiting por sessao (Map em memoria)
  • Loop com maxSteps=10 e deteccao de loop
  • Salva notas diarias automaticamente
  • Contexto MONTADO MANUALMENTE (sem ContextEngine)
DADOS core/session/transcript.ts

Session Transcript

Persistencia do historico de conversa em arquivos JSONL no disco (.colabor-ai/sessions/). Substitui o historico em RAM.

Funcoes: appendToTranscript, loadSessionTranscript, getRecentMessages, compactTranscript, generateSessionId. Inspirado no sistema de sessoes do OpenClaw.
NOVO core/skills/skills-manager.ts

SkillsManager

Sistema que carrega skills sob demanda. Le arquivos .md da pasta skills/ e filtra por relevancia ao contexto atual usando scoring por palavras-chave.

Status: 100% funcional! 3 skills criadas (communication, python-dev, web-search). Cache de 30s.
STREAM core/stream/event-stream.ts

EventStream

Sistema de streaming assincrono baseado em AsyncIterable. Permite que eventos sejam emitidos em tempo real durante a execucao do agente.

Tipos de eventos: agent_start/end, turn_start/end, text_start/delta/end, tool_call_start/delta/end. [SUBGratilizado] Estrutura existe mas eventos reais nao sao emitidos no loop.
EXECUCAO core/tools/

4 Ferramentas (Tools)

  • pythonExecTool — Executa Python em subprocesso
  • shellExecTool — Comandos shell com bloqueio de comandos perigosos
  • browserExecTool — Automacao via Puppeteer (navegar, clicar, screenshot)
  • task.tools — CRUD de tarefas (JSON file)
Cada tool segue o formato OpenAI function calling com schema de parametros e handler assincrono.
SUPORTE core/constants/ + core/utils/

Infraestrutura

  • constants/instructions.ts — Instrucoes de identidade e comportamento (CORE_IDENTITY + CORE_INSTRUCTIONS + FORMAT_RESPONSE_JSON)
  • utils/logger.ts — Logger estruturado leve (debug, info, warn, error) com niveis configuráveis via LOG_LEVEL
O CORE_IDENTITY define a personalidade do assistente: "You are not a chatbot. You are becoming someone."

5 Problemas Arquiteturais Identificados

# Problema Gravidade Onde Impacto
1 Context Engine incompleto Medio core/context/ Summarize stub, nao integrado ao loop
2 Agentes singleton Alto core/agents/*.ts Historico vaza entre sessoes de usuarios
3 Dupla definicao de agentes Medio main.ts vs agents/ Inconsistencia entre canais
4 Nomenclatura inconsistente Baixo answer.agent.ts AnswerAgent = WriterAgent = ConsoleAgent
5 Planner fragil (sem Zod) Alto orchestrator.ts:157 JSON mal formatado quebra o fluxo
6 Sem try/catch no loop Alto orchestrator.ts Excecao de agente derruba o fluxo
7 Modelo fixo sem fallback Alto agent.ts:41 Se DeepSeek cair, sistema para
8 Rate limit volatil Medio orchestrator.ts:12 Map em memoria zera ao reiniciar
9 Sem testes de integracao Medio geral Ciclo planner-agente-finish nao testado

Top 3 Mais Criticos

CRITICO

Planner sem validacao

JSON.parse() sem schema. LLM pode devolver JSON invalido e quebrar o fluxo.

CRITICO

Sem try/catch no loop

Se um agente lancar excecao, o fluxo inteiro quebra sem recuperacao.

CRITICO

Modelo fixo sem fallback

DeepSeek hardcoded. Se a API ficar indisponivel, sistema para.

6 Recomendacoes de Melhoria

URGENTE Semana 1-2
  • 1
    Model failover

    Array de modelos com fallback automatico. Se DeepSeek falha, tentar OpenAI.

  • 2
    Fila por sessao

    Evitar que duas mensagens do mesmo usuario executem ao mesmo tempo.

  • 3
    Event streaming detalhado

    Usar EventStream existente para emitir eventos reais durante execucao.

MEDIO Semana 3-4
  • 4
    Sumarizacao LLM no ContextEngine

    Completar o modo summarize com chamada LLM real. Integrar ao Orchestrator.

  • 5
    Hooks before/after tool

    Permitir validar/modificar chamadas de ferramentas antes da execucao.

  • 6
    Compromissos inferidos

    Apos cada resposta, extrair follow-ups implicitos e criar lembretes.

7 Pontos Fortes da Arquitetura Atual

Codigo modular

Separacao clara entre agentes, tools, e infraestrutura. Facil de estender.

Skills sob demanda

SkillsManager carrega apenas skills relevantes ao contexto, sem poluir o prompt.

EventStream

Estrutura de streaming em tempo real via AsyncIterable para frontends.

Transcript persistente

Conversas salvas em JSONL no disco. Nao perde historico entre execucoes.

Memoria de longo prazo

MEMORY.md + notas diarias + busca textual. 3 camadas de memoria persistentes.

Logger estruturado

Logger leve com niveis (debug, info, warn, error) e metadados.

Testes unitarios

Cobertura basica para Agent, Orchestrator, ContextEngine, SkillsManager.

Protecao contra loop

Deteccao de instrucao repetida e maxSteps=10 no Orchestrator.

Arquitetura inspirada

Baseada em conceitos do OpenClaw: ContextEngine, Skills, Session Transcript, Memory Search.
